Webhook Aggregator & DLQ Manager

An async event-processing pipeline with a real-time dashboard for inspecting and replaying dead-lettered webhook events — built to explore what production-grade reliability infrastructure actually looks like from the ground up.

⌥ GitHub

Why I Built This

I had a habit of placing myself in hypothetical engineering scenarios and asking: if I were running a platform that processed thousands of webhook events per hour, what would break first? The answer, almost always, is delivery. Webhooks fail — the downstream service is down, the network times out, the payload is malformed. What happens to those events?

In production systems, the answer is a Dead Letter Queue: a holding area for messages that couldn't be processed, with tooling to inspect what went wrong and replay the events once the underlying issue is resolved. I wanted to build one properly — not a toy version, but something with the moving parts a real system would have: async processing, persistent storage, retry logic, and a dashboard that gives an operator actual visibility.

What It Does

Incoming webhooks hit a FastAPI endpoint and are immediately pushed onto a RabbitMQ queue rather than processed inline. Celery workers pick up jobs from the queue asynchronously, attempt delivery, and handle failures — routing undeliverable events into the dead letter queue with their failure reason and metadata preserved. PostgreSQL backs the persistent state: event history, retry counts, timestamps, and current status.

The dashboard gives you a live view of the DLQ — what's in it, why it failed, and the ability to replay individual events or bulk-replay by failure type. The goal was to make the failure state of the system as visible and actionable as the happy path.

The interesting engineering constraint: replay has to be idempotent. Sending a webhook twice to a downstream service shouldn't cause a double-charge or a duplicate record. Building the retry logic to handle this correctly was the most thought-provoking part of the project.

Architecture

The pipeline has three distinct layers. The ingestion layer (FastAPI) accepts and validates incoming events and queues them immediately — keeping the HTTP response time fast regardless of downstream load. The processing layer (Celery + RabbitMQ) handles delivery attempts, exponential backoff on failure, and DLQ routing. The persistence layer (PostgreSQL) records the full event lifecycle so nothing is ever silently lost.

The dashboard reads directly from the database, giving operators a consistent view of system state without coupling the UI to the queue itself.

What I Learned

Building this made the distributed systems literature feel much more concrete. Concepts like at-least-once delivery, idempotency keys, and backpressure stopped being abstractions and became actual design decisions I had to make. I also came away with a much better intuition for when a message queue is the right tool — and when it's overkill.
